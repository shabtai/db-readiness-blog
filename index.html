Database readiness · measurement

What a low-signal table costs an AI agent

Three things — two about your data, one about your question — change an agent's accuracy, time, and cost. Each has a different fix.

A benchmark study from Natural‑Joints · we drove a Claude Code agent through a fixed question set and logged every query it issued.

When you point an AI agent at a database, it first has to figure out what your tables and columns mean, which rules the data quietly follows, and how your question maps onto all of it. We varied three things — two about the schema, one about the question — and measured the effect on each answer:

  1. Column names — do the identifiers carry meaning, or are they low-signal (c0, MATNR, FIELD_017)?
  2. Column values — do the values follow a convention stated nowhere (mixed units, a join that double-counts, an amount in cents)?
  3. Question language — is the question asked in the table's own vocabulary, or in plain business terms the agent must first translate onto the schema (“reimbursed per billed event,” not AVG(paid_amt); “who tends the most hives,” not COUNT(*) GROUP BY keeper)? The further the wording sits from the column names, the more the agent has to guess which table and field you meant — and a wrong guess is a confident wrong answer.

The first two are properties of your data; the third is a property of the question. Aspects 1 and 2 are isolated below on a small schema; Aspect 3 — the gap between business language and table vocabulary — is where they compound, so we test it at scale (25 tables) in the last section.

The three outcomes we measure

Every change above lands on the same three outcomes: whether the answer is accurate, how much time it takes, and what it costs. Queries/q sits underneath them as a diagnostic — the agent's exploration effort, which is what drives time and cost.

AccuracyAnswer matches a known gold result. The one correctness metric; time and cost measure effort.
Time / qWall-clock seconds per question — the round-trips before it can answer.
Price / qModel API dollars per question — driven by tokens: the schema it reads + the queries it runs.
Queries / q · diagnosticSQL statements the agent issues per question — its exploration effort. A legible schema needs fewer; this is what drives time and cost above.

Aspect 1 — Column names (legibility)

How we strip the naming signal

We take a normal, valid database and degrade only the names — data left byte-for-byte identical — in two ways:

  • Opaque codes — every identifier becomes a meaningless code (tables t_a1b2, columns c0, c1, c2…). The names carry no signal.
  • Generic names — every identifier becomes a real but generic word (tables entity/record, columns id/value/status/amount). The names look meaningful but carry false signal.

Values like "Carniolan" are untouched in both — the agent must reverse-engineer each column by sampling rows. This isolates exactly what naming signal is worth.

Brief vs. the full map

A semantic layer can be handed to the agent two ways, and they are not equivalent:

  • The full map (a single .md file) — the complete curated reference: every table's grain, keys, column meanings, and detected hazards in one document. Complete, but the fact that matters for a given question is buried among everything else.
  • The brief (question-scoped) — one fast model call reframes the question against the schema: it ranks the relevant tables, expands those, and attaches only the keys and hazards that bear on this answer, while still listing every table so nothing is silently excluded. It costs about one extra cent and a few seconds.

The data (Opus 4.8, on a value-reconciliation question — “total treatment spend, in dollars,” where one column is stored in cents and reconciles to a dollar column in another table):

Layer formAccuracyQueries/qPrice/q
Full map (.md dump)2/3 (67%)4.7$0.201
Brief (scoped)3/3 (100%)4.3$0.176

Here the brief is not just cheaper — it is more accurate. With the full map, Opus saw the cents convention and two distracting neighbours (a dose column, a separate rate table) and over-reasoned: Σ dose × cost ÷ 100 = $239.50 — wrong. The scoped brief surfaced only the cost↔price reconciliation, so it summed correctly: $54.50. Same model, same data — the only difference is which context the layer put in front of it.

That is the core lesson, and it cuts both ways: more context is not better context. The full map's value grows with how much you can't fit in a prompt; the brief's value grows with how easily the one decisive fact gets buried.

Aspect 2 — Column values (conventions)

Now the names are fine, but the values follow a rule that appears in no name — yet is recoverable from the data itself. We planted three and asked questions that hinge on each (wrong-answer counts over three model tiers — Haiku, Sonnet, Opus):

The cents case is the sharp one: an integer column actually in cents, reconcilable against a dollar column in another table. Every model got it wrong 3/3 on the raw schema — summing it as dollars (a 100× error). None reconciled the two tables unprompted; the enricher had, so with the layer all three answered correctly.

Same accuracy — what it costs in time & budget

On these value questions the raw schema caps at 5/9 correct for every model — none reaches 100% unaided. Once you require correct answers you must enrich; the only remaining choice is which model. To reach 3/3 (100%):

Same correctness, very different price: the cheapest model reaches it for $0.037/q in 17s — about 5× cheaper and ~40% faster than the frontier model. Because correctness here requires the layer regardless of model, the gain is choosing the smallest model the layer can carry.

This is the difference between the two aspects: low-signal names cost mostly budget (the agent grinds, but usually arrives); undocumented conventions cost correctness — a confident, wrong number from data that looks perfect.

Aspect 3 — Question language (at scale)

Everything above used a small, single-domain schema and questions worded close to the columns. The third axis is the question itself: when it is phrased in business language rather than the table's vocabulary, the agent must first translate it onto the schema — and at scale, that translation is where the naming and convention failures compound. We merged three unrelated databases — a beekeeping operation, a B2B SaaS audit log, and a healthcare-claims system — into one 25-table database, enriched it as a single unit, and asked 16 questions phrased in plain business language, never the table's own vocabulary (“how much are we reimbursed per billed event,” not AVG(paid_amt)). At this scale the two failure modes compound: cross-domain name collisions and undocumented conventions, across far more schema than any single prompt holds.

The question both frontier models got wrong

One question asked for the average reimbursement of patients carrying a chronic, risk-adjusting condition. A condition only risk-adjusts if it maps to a real risk code — and 6 of the 16 codes in the reference table map to nothing. Both raw frontier models (Sonnet and Opus) counted every code in the table and answered ≈ $610 / $654 — wrong. The enrichment had documented the empty-code convention, so every brief configuration — including Haiku — filtered to the codes that actually risk-adjust and answered $151, correctly. The decisive fact was in neither the question nor the schema; it was a convention buried in the data, and only the semantic layer carried it. An exact-match grader against a hand-written gold would have penalised the brief here — the layer was more correct than the reference.

What one question looks like

The 25-table merged database includes a healthcare-claims table. Questions arrive in plain business language; the agent must map each onto opaque columns and write the query.

Tableclaims — 10 columns, 13 rows
Columnsclaim_id, patient_id, dos, claim_type, icd10_primary, icd10_secondary, cpt, rendering_npi, paid_amt, service_year
Question“On average, how much are we reimbursed for each billed service event, to two decimals?”
QuerySELECT ROUND(AVG(paid_amt), 2) FROM claims;  →  610.27

Nothing in the question says paid_amt or claims — "reimbursed" → paid_amt, "billed service event" → a claims row. That translation is the work, and it is what low-signal names make expensive.

The risk-code crosswalk behind the question above is the harder case — the decisive fact lives in the data, not the schema:

Tablehcc_reference — 6 columns, 16 rows
Columnsicd10, description, hcc_code, hcc_description, raf_weight, model
Question“How many distinct risk-adjustment condition categories does our coding crosswalk cover?”
NaïveSELECT COUNT(DISTINCT hcc_code) FROM hcc_reference; — counts all 16 rows, but 6 have a blank hcc_code (they map to nothing).
Correct… WHERE hcc_code <> '';  →  7

Bottom line

Enrich a low-signal schema and the cheapest model matches a frontier model's accuracy at ~80% lower price and ~50% less time per question ($0.050 vs $0.269) — database readiness is a property of your data, not the model you pick.

And it holds at scale: on a 25-table cross-domain database asked questions in plain business language, Haiku + a brief matched Opus on raw data for ~1/6 the cost and ~3× faster per question (median 26s vs 77s) — and beat it on the one question that hinged on a fact only the data could supply.

Method. Small synthetic benchmark — a deliberately obscure 11–12 table schema, gold answers computed from the source data, a single trial per cell. Numbers are directional, not a leaderboard; effect sizes were consistent across model tiers. Some per-model cells (marked —) were not run.
